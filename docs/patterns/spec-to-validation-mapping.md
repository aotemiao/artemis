# Spec To Validation Mapping

Status: maintained
Last Reviewed: 2026-06-01
Review Cadence: 90 days

本模式用于把 Feature Spec 的验收标准映射到可执行验证入口，避免“需求写清楚了，但没有证明实现满足需求”。

## 映射原则

每条验收标准都应至少对应一种验证方式：

- 单元测试：适合领域规则、值对象、App 层用例。
- 集成测试：适合持久化、Gateway 实现、Controller 行为。
- smoke：适合关键端到端链路和本地服务可用性。
- harness 脚本：适合文档、契约、目录结构、质量门。
- 人工验收：只用于 UI 视觉判断、外部系统不可自动化场景或一次性业务确认。

## 推荐表格

```md
| 验收编号 | 验证入口 | 通过标准 |
|----------|----------|----------|
| AC-001 | `scripts/harness/check-api-doc-sync.sh` | 新增接口均已在 API 文档登记 |
| AC-002 | `mvn test -pl <module> -am` | 领域规则测试通过 |
| AC-003 | `scripts/smoke/<name>.sh` | 关键链路返回期望状态码和响应字段 |
```

## 审核重点

- 验证入口必须真实存在。
- `通过标准` 不能只写“通过”，应说明关键断言或期望输出。
- 如果使用人工验收，必须说明为什么不能自动化，以及验收人需要检查什么。
- 如果一条验收标准没有验证方式，应先补测试、脚本或 smoke，再进入实现收尾。
