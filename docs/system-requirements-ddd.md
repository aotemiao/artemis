# Artemis 功能需求梳理

> 目标：把当前系统已经实现的业务能力整理成可用于 DDD 重构的新系统需求输入。  
> 范围：以后端源码、初始化 SQL 菜单、数据表和服务规则为依据；前端页面仅按菜单和接口能力推断，不展开 UI 设计细节。
> 补充验收清单：[requirements-acceptance-checklist.md](./requirements-acceptance-checklist.md)

## 1. 系统定位

本系统是一个面向多租户后台管理场景的微服务权限管理平台，核心能力包括：

- 多租户 SaaS 管理：租户、租户套餐、租户域名、租户有效期、用户额度、租户初始化。
- 身份认证与会话：验证码、账号密码、短信、邮箱、第三方授权、小程序登录、注册、登出、在线会话管理。
- 组织与权限：用户、部门、岗位、角色、菜单、按钮权限、数据权限、客户端授权配置。
- 系统基础数据：字典、参数、通知公告、社会化账号绑定关系。
- 审计与监控：操作日志、登录日志、缓存监控、在线用户、账户解锁、外部监控控制台入口。
- 资源服务：对象存储文件管理、对象存储配置、短信、邮件、站内消息。
- 工作流：流程分类、流程定义、流程设计、流程实例、我的任务、流程监控、任务办理、请假示例业务。
- 开发工具：数据库表导入、代码生成配置、预览、下载、同步数据库。
- 技术示例：单表/树表 CRUD、Excel、Redis、ES、MQ、短信、邮件、权限校验、加解密、脱敏、分库分表等演示能力。

## 2. 主要用户角色

| 角色 | 说明 | 典型能力 |
| --- | --- | --- |
| 超级管理员 | 平台级管理员，默认管理租户下拥有最高权限 | 管理租户、套餐、所有系统配置、动态切换租户、查看全部权限 |
| 租户管理员 | 每个租户初始化时创建的管理员 | 管理本租户用户、部门、岗位、角色、菜单范围内的业务 |
| 系统管理员/运维人员 | 负责平台运行配置和监控 | 参数、字典、客户端、日志、在线用户、缓存、外部控制台 |
| 组织管理员 | 负责组织架构和人员维护 | 部门、岗位、用户、角色授权 |
| 普通后台用户 | 使用后台业务功能 | 登录、个人资料、查看授权菜单、发起和处理流程 |
| 流程设计人员 | 负责流程模板配置 | 流程分类、流程定义、节点、条件、办理人、发布/停用 |
| 流程审批人员 | 处理流程任务 | 待办、已办、抄送、审批、退回、转办、委派、加减签、终止 |
| 审计人员 | 查看系统行为记录 | 操作日志、登录日志、导出、清理 |

## 4. 功能需求

### 4.1 身份认证与会话

#### AUTH-001 图形验证码

- 系统应支持生成图形验证码，返回是否启用验证码、验证码 UUID 和验证码图片。
- 验证码应存入缓存，并在校验后删除，避免重复使用。
- 登录和注册在验证码开关开启时应校验验证码，验证码不存在视为过期，内容不一致视为错误。

#### AUTH-002 登录

- 登录请求应包含客户端 ID、授权类型、租户 ID，以及对应登录方式所需参数。
- 系统应先校验客户端是否存在、是否支持当前授权类型、客户端状态是否正常。
- 多租户开启时，系统应校验租户编号不能为空；非默认租户必须存在、未停用、未过期。
- 系统应支持以下登录方式：
  - 账号密码登录：用户名、密码、图形验证码。
  - 短信登录：手机号、短信验证码。
  - 邮箱登录：邮箱、邮箱验证码。
  - 第三方登录：第三方平台、授权 code、state。
  - 小程序登录：appid、小程序 code。
- 登录成功后应返回 access token、refresh token、过期时间、client_id、scope、openid 等令牌信息。
- 登录失败应记录登录日志，并按登录类型累计错误次数；超过阈值后应在配置的锁定时间内拒绝登录。
- 登录成功后应清除该账号登录失败次数。
- 登录成功后系统可异步发送欢迎消息。

#### AUTH-003 登出

- 已登录用户应可主动登出。
- 登出应记录登录日志。
- 超级管理员登出时，如启用了动态租户切换，应清除动态租户上下文。
- 无登录态时调用登出不应造成业务异常。

#### AUTH-004 用户注册

- 系统应通过参数配置控制是否允许注册。
- 注册请求应包含租户 ID、客户端 ID、授权类型、用户名、密码、用户类型，以及可选验证码。
- 注册时应校验用户类型合法性。
- 新用户密码应加密存储。
- 注册成功/失败应记录登录日志。

#### AUTH-005 第三方账号绑定与解绑

- 系统应支持按第三方登录来源生成授权地址，并携带租户 ID、域名和 state。
- 第三方回调成功后，应绑定当前登录用户与第三方账号。
- 同一第三方账号 authId 已经被绑定时，不允许重复绑定。
- 同一用户在同一第三方来源已有绑定时，允许更新绑定信息。
- 用户应能解绑自己的第三方授权关系。

#### AUTH-006 登录租户列表

- 登录页应能查询可选租户列表。
- 多租户未启用时，应直接返回“未启用多租户”标记。
- 超级管理员可看到所有租户。
- 普通访问应优先按请求域名筛选租户；没有匹配时返回全部可见租户。
- 该接口应按 IP 限流。

### 4.2 租户管理

#### TENANT-001 租户套餐管理

- 系统应支持租户套餐的分页查询、下拉选择、详情、创建、修改、启停、删除和导出。
- 租户套餐应包含套餐名称、可用菜单 ID 集合、菜单父子联动选择规则、状态和备注。
- 套餐名称应唯一。
- 已被租户使用的套餐不允许删除。
- 套餐状态可启用/停用。

#### TENANT-002 租户管理

- 系统应支持租户的分页查询、详情、创建、修改、启停、删除、导出。
- 租户信息应包含联系人、联系电话、企业名称、统一社会信用代码、地址、域名、简介、套餐、有效期、用户数量额度、状态和备注。
- 企业名称应唯一。
- 创建租户时系统应自动生成 6 位租户编号，且不得与已有租户重复。
- 创建租户时应基于租户套餐自动创建租户管理员角色，并为该角色授予套餐中的菜单权限。
- 创建租户时应自动创建根部门，部门名称默认为企业名称。
- 创建租户时应自动创建首个系统用户，并将其设为根部门负责人。
- 创建租户时应建立首个用户与租户管理员角色的关系。
- 创建租户时应从默认租户复制字典类型、字典数据、系统参数到新租户。
- 创建租户时应同步默认工作流定义到新租户。
- 修改租户时不允许直接修改租户编号和套餐 ID，应通过套餐同步功能处理套餐变更。
- 默认管理租户不允许被普通租户操作。
- 超级管理员租户不允许删除。

#### TENANT-003 租户运行约束

- 租户停用后，用户不得登录该租户。
- 租户过期后，用户不得登录该租户。
- 用户额度为 -1 表示不限制用户数量；否则新增用户时应检查剩余额度。
- 超级管理员可动态切换租户，并可清除动态租户。
- 系统应支持手动同步租户套餐、租户字典和租户参数配置。

### 4.3 用户、组织与权限

#### IAM-001 用户管理

- 系统应支持用户分页查询、详情、创建、修改、删除、导入、导出和导入模板下载。
- 用户应包含租户、部门、账号、昵称、用户类型、邮箱、手机号、性别、头像、密码、状态、登录 IP、登录时间、岗位、角色和备注。
- 用户账号、手机号、邮箱应分别保持唯一。
- 创建和修改用户时应维护用户与岗位、角色的关联关系。
- 普通用户不允许被分配超级管理员角色。
- 如果角色或岗位不在当前操作者数据权限范围内，不允许分配。
- 不允许操作超级管理员用户。
- 删除用户时应同步删除用户角色和用户岗位关系。
- 用户状态可启用/停用。
- 管理员应能重置用户密码。
- 系统应能按部门查询用户、按用户 ID 批量查询基础信息。

#### IAM-002 个人中心

- 当前用户应能查看个人资料，包括基础信息、所属角色、所属岗位。
- 当前用户应能修改昵称、手机号、邮箱、性别等个人资料。
- 当前用户应能修改登录密码。
- 当前用户应能上传并更新头像。
- 当前用户应能查看自己的在线设备，并强退指定设备。

#### IAM-003 部门管理

- 系统应支持部门树查询、排除指定节点及其子节点的查询、详情、创建、修改、删除和下拉选择。
- 部门应包含父部门、祖级列表、部门名称、部门类别、排序、负责人、电话、邮箱、状态。
- 同一父部门下部门名称应唯一。
- 新增部门时，父部门必须处于正常状态；停用部门下不允许新增子部门。
- 修改部门父级时，应同步更新所有子部门的祖级列表。
- 启用某部门时，应自动启用其上级部门。
- 非超级管理员只能访问数据权限范围内的部门。

#### IAM-004 岗位管理

- 系统应支持岗位分页查询、详情、创建、修改、删除、导出、下拉选择和部门树查询。
- 岗位应包含所属部门、岗位编码、岗位类别、岗位名称、排序、状态和备注。
- 同一部门下岗位名称应唯一。
- 岗位编码应唯一。
- 已分配给用户的岗位不允许删除。

#### IAM-005 角色管理

- 系统应支持角色分页查询、详情、创建、修改、删除、导出、启停、下拉选择。
- 角色应包含名称、权限字符串、排序、数据范围、菜单树联动规则、部门树联动规则、状态和备注。
- 角色名称应唯一。
- 角色权限字符串应唯一。
- 不允许操作超级管理员角色。
- 不允许新增或非法修改系统内置管理员角色标识符。
- 已分配给用户的角色不允许删除。
- 已分配给用户的角色不允许禁用。
- 系统应支持角色菜单授权。
- 系统应支持角色数据权限设置，并维护角色与部门关系。
- 系统应支持查看已分配/未分配某角色的用户。
- 系统应支持给角色批量授权用户、批量取消授权用户、取消单个用户授权。
- 不允许修改当前登录用户自己的角色授权。
- 角色授权变化后，应清理相关在线用户登录状态，使权限变更及时生效。

#### IAM-006 菜单和按钮权限

- 系统应支持菜单列表、详情、创建、修改、删除、批量级联删除。
- 菜单类型包括目录、菜单、按钮。
- 菜单应包含名称、父级、排序、路由地址、组件路径、路由参数、外链标记、缓存标记、可见状态、启停状态、权限标识、图标、备注。
- 同一父级下菜单名称应唯一。
- 非按钮菜单的路由配置应唯一。
- 系统应根据当前用户权限生成前端路由。
- 系统应支持菜单树下拉、角色菜单树、租户套餐菜单树。
- 批量级联删除菜单时应同步删除角色菜单关联。

#### IAM-007 客户端管理

- 系统应支持客户端分页查询、详情、创建、修改、删除、导出和启停。
- 客户端应包含 client_id、client_key、client_secret、授权类型、设备类型、token 活跃超时、token 固定超时、状态。
- 登录时应校验客户端存在、状态正常，并且授权类型在客户端允许范围内。

### 4.4 系统基础数据

#### SYS-001 字典管理

- 系统应支持字典类型分页查询、详情、创建、修改、删除、导出、下拉选择和刷新缓存。
- 系统应支持字典数据分页查询、详情、创建、修改、删除、导出，并可按字典类型获取字典数据。
- 字典类型编码应唯一。
- 删除字典类型前应检查是否存在字典数据；已分配数据的字典类型不允许删除。
- 修改字典类型编码时，应同步更新其字典数据的字典类型。
- 字典数据应支持排序、标签、键值、样式类、回显样式、默认标记和备注。

#### SYS-002 参数配置

- 系统应支持参数分页查询、详情、创建、修改、按 key 修改、删除、导出、刷新缓存。
- 参数应包含参数名称、参数键名、参数键值、是否系统内置和备注。
- 参数键名应唯一。
- 系统内置参数不允许删除。
- 系统应支持按参数 key 查询参数值。
- 参数配置应被缓存，并在增删改和刷新时维护缓存一致性。

#### SYS-003 通知公告

- 系统应支持通知公告分页查询、详情、创建、修改、删除。
- 公告应包含标题、类型、内容、状态和备注。

#### SYS-004 社会化关系

- 系统应支持查询当前用户或系统内的社会化账号绑定关系列表。
- 绑定关系应记录用户、租户、第三方来源、openId、unionId、昵称、头像、授权 token 等信息。

### 4.5 审计、日志与在线用户

#### AUDIT-001 操作日志

- 系统应自动记录后台操作日志。
- 操作日志应包含模块标题、业务类型、方法、请求方式、操作类别、操作人、部门、URL、IP、地点、请求参数、返回结果、状态、错误信息、耗时、操作时间。
- 系统应支持操作日志分页查询、导出、批量删除和清空。

#### AUDIT-002 登录日志

- 系统应记录登录、登出、注册、登录失败等访问记录。
- 登录日志应包含租户、用户账号、客户端、设备类型、IP、地点、浏览器、操作系统、状态、消息和登录时间。
- 系统应支持登录日志分页查询、导出、批量删除、清空。
- 系统应支持账户解锁，清除登录失败锁定状态。

#### AUDIT-003 在线用户

- 系统应支持按 IP 和用户名查询在线用户。
- 系统应展示在线用户的 token、用户信息、登录信息和会话信息。
- 管理员应能强退指定在线用户。
- 用户应能查看自己的在线设备，并强退自己的指定设备。

#### AUDIT-004 缓存监控

- 系统应支持查看 Redis/缓存监控信息。
- 缓存监控菜单应对具备 `monitor:cache:list` 权限的用户可见。

### 4.6 资源、文件与消息

#### RES-001 对象存储文件管理

- 系统应支持 OSS 文件分页查询、按 ID 列表查询、上传、下载、删除。
- 文件信息应包含文件名、原始文件名、后缀、URL、上传人、服务商和扩展字段。
- 上传文件不能为空。
- 下载不存在的文件应返回业务错误。
- 删除文件时应删除文件记录，并尝试删除对象存储中的实际对象。
- 系统应提供远程文件服务，用于其他服务上传文件、根据 ID 查询 URL 和文件列表。

#### RES-002 对象存储配置

- 系统应支持对象存储配置分页查询、详情、创建、修改、删除、启停/默认状态修改。
- 配置应包含配置 key、accessKey、secretKey、bucket、前缀、endpoint、自定义域名、HTTPS、region、访问策略、状态、服务商扩展字段。
- 配置 key 应唯一。
- 系统内置配置不允许删除。
- 状态修改应能切换默认/启用配置。

#### RES-003 短信、邮箱与站内消息

- 系统应支持发送短信验证码和邮箱验证码。
- 系统应提供远程短信服务，支持单发、批量群发、模板消息、异步发送、延迟发送、黑名单添加和移除。
- 系统应提供远程邮件服务，支持发送邮件。
- 系统应提供远程站内消息服务，支持向指定用户发布消息和全员发布消息。
- 工作流和登录欢迎消息可复用消息服务。

### 4.7 工作流

#### WF-001 流程分类

- 系统应支持流程分类分页查询、详情、创建、修改、删除、导出和分类树查询。
- 流程分类应支持父子层级、祖级列表、名称、排序。
- 新增或修改分类时，父级分类必须存在。
- 顶级分类不允许修改父级节点。

#### WF-002 流程表达式

- 系统应支持流程 SpEL 表达式列表、详情、创建、修改、删除。
- 表达式应包含组件名称、方法名、参数、预览表达式、备注和状态。
- 同一预览表达式不允许重复添加。
- 初始表达式包括按部门 ID 获取部门负责人、流程发起人变量等。

#### WF-003 流程定义

- 系统应支持流程定义分页查询、未发布列表、详情、创建、修改、删除、复制、发布、取消发布、导入、导出、查看 JSON/XML 字符串、激活/挂起。
- 流程定义应包含流程编码、流程名称、模型类型、流程分类、版本、发布状态、自定义表单标记、表单路径、激活状态、监听器、扩展 JSON、租户。
- 发布流程定义前，应校验中间节点配置了办理人；未配置办理人的节点不允许发布。
- 已被流程实例使用的流程定义不允许删除。
- 只有已发布的流程定义才能启动流程。
- 流程定义可被复制为新的定义。
- 系统应支持将流程定义同步到新租户。

#### WF-004 流程启动

- 启动流程必须提供业务 ID 和流程编码。
- 系统应将流程发起人、发起人部门、业务 ID 写入流程变量。
- 系统应保存流程实例业务扩展信息，包括业务编码、业务标题、业务 ID、实例 ID。
- 如果同一业务 ID 已存在流程实例，应校验当前状态：
  - 待审核、已完成、已作废、已终止、状态为空时不允许重新启动。
  - 草稿、已撤销、已退回状态可继续提交。
- 如果流程编码未找到已发布定义，应拒绝启动。
- 启动后必须生成且只生成一个申请人初始任务，否则视为流程定义配置错误。

#### WF-005 任务办理

- 系统应支持办理任务，办理时可提交审批意见、附件、流程变量、下一节点办理人、抄送人、消息类型和消息通知内容。
- 任务不存在或已审批时不允许办理。
- 流程实例不存在时不允许办理。
- 草稿、已撤销、已退回状态的流程在再次办理时应识别为重新提交。
- 办理通过后应按流程定义生成后续任务和历史任务。
- 消息类型包括站内信、邮箱、短信。

#### WF-006 我的任务与流程监控

- 系统应支持查询当前用户待办任务。
- 系统应支持查询当前用户已办任务。
- 系统应支持查询当前用户抄送任务。
- 系统应支持查询全部待办任务和全部已办任务，用于流程监控。
- 系统应支持查询当前用户发起的流程实例。
- 系统应支持查询运行中的流程实例和已结束流程实例。
- 系统应支持按业务 ID 查询流程实例详情。
- 系统应支持查询流程图和流程审批历史。
- 系统应支持查询当前任务详情，包括按钮权限、可抄送人员、变量、节点比例、是否申请人节点。

#### WF-007 流程实例管理

- 系统应支持按业务 ID 删除流程实例。
- 系统应支持按实例 ID 删除运行中流程实例。
- 系统应支持按实例 ID 删除已完成历史流程实例。
- 删除流程实例时，只有创建人或管理员可操作。
- 系统应支持撤销流程；非创建人且非管理员不允许撤销。
- 撤销前应校验流程未完成、未作废、未终止、未退回、未撤销。
- 系统应支持流程实例激活/挂起。
- 系统应支持查询和修改流程变量。
- 系统应支持作废流程；已完成、已作废、已终止或状态为空不允许作废。

#### WF-008 任务操作

- 系统应支持终止任务。
- 系统应支持驳回审批，并可查询可驳回的前置节点。
- 系统应支持委派任务。
- 系统应支持转办任务。
- 系统应支持加签和减签；仅会签或票签节点允许加减签。
- 系统应支持批量修改任务办理人。
- 系统应支持查询当前任务所有办理人。
- 系统应支持催办任务。
- 超级管理员或租户管理员执行任务操作时可忽略部分任务归属限制。

#### WF-009 流程状态

业务流程状态应至少包括：

| 状态码 | 描述 | 说明 |
| --- | --- | --- |
| `draft` | 草稿 | 初始或待提交 |
| `waiting` | 待审核 | 审批中 |
| `finish` | 已完成 | 流程正常完成 |
| `cancel` | 已撤销 | 发起人或管理员撤销 |
| `back` | 已退回 | 审批被退回 |
| `invalid` | 已作废 | 管理操作作废 |
| `termination` | 已终止 | 任务被终止 |

任务历史状态应至少包括：撤销、通过、待审核、作废、退回、终止、转办、委托、抄送、加签、减签、超时。

#### WF-010 节点办理人和按钮权限

- 节点办理人来源应支持用户、角色、部门、岗位、SpEL 表达式。
- SpEL 表达式应支持 `$` 和 `#` 前缀的变量/表达式策略。
- 节点按钮权限应支持：弹窗选人、委托、转办、抄送、退回、加签、减签、终止。

#### WF-011 请假示例业务

- 系统包含请假申请示例，支持列表、详情、创建、修改、删除、导出。
- 请假单应包含申请编号、请假类型、开始时间、结束时间、请假天数、请假原因、状态。
- 请假单可提交并启动工作流。
- 请假流程事件应回写请假单状态。
- 请假示例属于演示业务，DDD 重构时可作为工作流接入样例，而不是平台核心领域。

### 4.8 代码生成

#### GEN-001 数据库表导入

- 系统应支持查询已导入代码生成列表。
- 系统应支持查询数据库表列表，并可选择表导入生成配置。
- 系统应支持按数据源查询表，支持查询当前可用数据源名称列表。

#### GEN-002 生成配置维护

- 系统应支持查看生成表详情。
- 系统应支持查看生成表字段列表。
- 系统应支持修改生成配置，包括包名、模块名、业务名、功能名、作者、模板类型、生成方式、生成路径、字段配置和查询/表单配置。
- 系统应支持删除生成配置。
- 系统应支持同步数据库结构到生成配置。

#### GEN-003 代码生成输出

- 系统应支持预览代码。
- 系统应支持下载 ZIP 代码包。
- 系统应支持按自定义路径生成代码。
- 系统应支持批量生成代码。

### 4.9 系统监控与运维集成

#### OPS-001 外部控制台入口

- 系统菜单应提供 SnailJob 控制台入口，用于分布式任务调度管理。
- 系统菜单应提供 Nacos 控制台入口，用于服务注册和配置管理。
- 系统菜单应提供 Spring Boot Admin 控制台入口，用于服务监控。

- 监控服务应能接收 Prometheus 告警消息。

#### OPS-003 分布式任务示例

- 任务模块包含 SnailJob 执行器示例，包括普通任务、广播任务、静态分片、Map/MapReduce，以及支付宝/微信账单和汇总任务示例。
- 任务调度管理主要依赖 SnailJob 控制台，当前业务模块不提供自建任务管理 Controller。

## 7. 跨领域规则和非功能需求

- 多租户隔离：业务表普遍包含 `tenant_id`，默认租户编号为管理租户；租户上下文需要贯穿认证、权限、配置、字典、文件和工作流。
- 访问控制：系统采用角色权限、菜单权限、按钮权限和数据权限共同控制访问。
- 数据权限：角色支持全部数据、自定义部门、本部门、本部门及以下、仅本人、部门及以下或本人等范围。
- 审计：新增、修改、删除、导入、导出、清理、授权、强退等操作应记录操作日志。
- 缓存：字典、参数、部门、角色、昵称等高频数据应缓存；修改后应清理或更新缓存。
- 幂等/并发：第三方绑定、流程启动、任务办理等关键操作应使用分布式锁防并发重复。
- 安全：登录和注册接口应支持请求体加密；密码应加密存储；输入应支持校验和 XSS 防护。
- 软删除：多张业务表包含 `del_flag`，新系统应明确软删除策略和唯一性约束是否排除已删除数据。
- 导入导出：用户、角色、岗位、字典、参数、租户、套餐、流程分类、流程定义、请假示例等支持 Excel 或文件导入导出。
- 国际化：校验消息和部分业务消息通过国际化 key 管理。
- 微服务协作：认证、系统、资源、工作流之间通过远程服务互相调用，DDD 重构时应显式定义上下文间防腐层/API。
