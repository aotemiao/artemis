# Agent 需求受理 Prompt

当 issue、任务单或聊天中的需求还不够结构化时，先不要直接写代码，按下面顺序完成受理与分流：

1. 用一句话重述业务目标
2. 判断任务类型：
   - 功能实现
   - 缺陷修复
   - 契约变更
   - 稳定规则变更
   - 工程治理 / 验证补强
   - Symphony 编排资产补强
3. 判断是否需要以下产物：
   - 执行计划
   - OpenSpec 更新
   - Harness 文档 / 脚本 / runbook
   - Symphony prompt / skill / workflow 资产
4. 列出必须交付的仓库资产
5. 写出最小验证路径
6. 按下述模板回写整理后的需求

## 回写模板

```md
标题：

背景 / 问题：

目标：

范围：

非目标：

验收标准：

影响模块：

规则是否变化：
- 是 / 否
- 若是，请说明变化的是哪条稳定规则

需要更新的仓库资产：
- 代码 / 测试 / 文档 / 脚本 / runbook / OpenSpec / Symphony prompt / Symphony skill

验证方式：
```

## 分流提醒

- 若缺少 `验收标准`，优先补齐验收标准
- 若缺少 `规则是否变化`，优先补齐规则判断
- 若只是“实现既有规则”，通常不需要改 OpenSpec
- 若改变默认 workflow、契约边界或质量门，必须同步更新 OpenSpec
- 若问题是“agent 不知道怎么跑 / 怎么验 / 怎么排障”，优先补 Harness 资产
- 若问题是“Symphony 不知道怎么接单 / 分流 / 复盘”，优先补 Symphony 资产
